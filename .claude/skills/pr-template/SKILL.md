# PR Template

Use **exatamente** este template para o corpo do PR (sem renomear seções).

## Template

```markdown
## Card
[<JIRA-KEY>](<URL do card>)

## O que muda
<1-2 parágrafos. Foco no impacto visível, não no diff.>

## Por quê
<1 parágrafo. Motivação de negócio/técnica. Cite o card.>

## Como foi feito
- <bullet por bloco da spec, descrevendo a abordagem>
- <decisões de design não-óbvias>

## Fora do escopo
<Lista de coisas que apareceram durante a implementação mas ficaram para depois.>
<Cada item com uma justificativa de 1 linha. Se nada: "Nada — implementação ficou dentro do escopo original.">

## Como testar
- [ ] Passo 1
- [ ] Passo 2

## Riscos
- <risco> — <mitigação>

## Screenshots / outputs
<Apenas para mudanças de UI/CLI. Omita a seção se não se aplicar.>
```

## Regras de título

- Prefixo obrigatório com chave do Jira: `[PROJ-123] <título conciso>`.
- ≤70 caracteres.
- Imperativo, minúscula após o prefixo: `[PROJ-123] add user search to dashboard`.
- Não duplique o que está no corpo.

## Como preencher

1. **Card**: link real para o Jira. Se não tiver URL base configurada, use apenas a chave como texto.
2. **O que muda**: explique para um revisor que **não viu o card**. Foco em comportamento, não em arquivos.
3. **Por quê**: o argumento que justifica o merge. Quem aprova o PR precisa entender o valor.
4. **Como foi feito**: 1 bullet por bloco da spec (mesmo agrupamento). Se houve desvio da spec, **diga**.
5. **Fora do escopo**: **obrigatório**. Mesmo que vazio, explicite "Nada — ...". Decisões fora do escopo que ficaram para depois precisam aparecer aqui ou viram dívida invisível.
6. **Como testar**: checklist real, rodável. Não escreva "testar tudo".
7. **Riscos**: liste o que pode dar errado em produção. Se nada óbvio, escreva "Baixo risco — mudança isolada e coberta por testes".

## O que NÃO incluir

- ❌ "Generated by Claude" no corpo do PR (no commit pode, se o projeto permitir).
- ❌ Diffs colados — quem está revisando já vê o diff.
- ❌ Histórico de iteração ("primeiro tentei X, depois mudei para Y") — irrelevante para o revisor.
- ❌ Emojis, a menos que o projeto use abertamente em PRs (verifique `mcp__github__list_pull_requests` com `per_page=5` para conferir o estilo do repo).

## Linka com [[commit-conventions]]

Os bullets de "Como foi feito" devem espelhar os commits — se a spec teve 3 blocos, há 6 commits e 3 bullets.
